## 一般工作流程

加入[我们深受喜爱的贡献者](https://github.com/browserbase/stagehand/graphs/contributors)行列！

1. **在开始前先讨论你的计划贡献。** 否则你可能会白白投入大量时间和精力。你可以在 [Slack](https://stagehand.dev/slack) 私信 Miguel 安排 1 对 1 通话。
2. **打开一个 Pull Request。** Fork 本仓库，并按照[GitHub 的 Pull Request 创建指南](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/proposing-changes-to-your-work-with-pull-requests/creating-a-pull-request-from-a-fork)操作。这样我们的团队就能审阅你的贡献并留下评论。 
3. **等待审核。** 我们会尽快处理你的贡献。如果已过 2–3 天仍未收到任何评论，请在 [Slack](https://stagehand.dev/slack) 私信 Miguel。
4. **合并到 `evals` 分支。** 为防止垃圾和滥用，我们不允许外部贡献者[通过 GitHub Actions 运行我们的 CI](https://github.com/browserbase/stagehand/blob/main/.github/workflows/ci.yml)。如果你的贡献通过初筛，我们会在其上运行我们的评估套件与仪表盘（用于比较 Stagehand 任务的模型性能与成本）
    1. 默认情况下，所有 PR 都会运行以下测试，你也可以在仓库根目录运行：
        1. Lint（`npm run lint`）— 运行 `prettier` 和 `eslint`。如果失败，通常可以运行 `npm run format` 修复一些简单的 Lint 错误。
        2. Build（`npm run build`）— 通过 `tsup` 在 `dist/` 中进行 TS → JS 的 Lint 与构建
        3. 端到端（`npm run e2e`）— 确定性的 Playwright 端到端测试，用于确保 [`stagehand.page`](http://stagehand.page) 和 `stagehand.context` 的基本 Playwright 功能完整性，以及与 Browserbase API 的兼容性
        4. 组合（`npm run evals category combination`）— 使用 `act`、`extract` 和 `observe` 的组合运行基于 AI 的端到端测试
    2. 如果你对 `act`、`extract` 或 `observe` 本身做了修改，我们也可能运行针对 act/extract/observe 的评估套件与仪表盘（用于比较 Stagehand 任务的模型性能与成本），以确保现有功能没有显著下降。
    
    ![CI](/images/CI.png)
    
5. **清理并合并到 main。** 一旦进入 `evals`，很遗憾原始贡献者将无法再做进一步更改。Stagehand 内部团队将负责清理代码并将其合并进 main。 

## 贡献指南

1. **使用草稿 PR。** 如果你的 PR 仍在进行中，请在处理期间将其转换为草稿（见下文），并在准备就绪时标记为可审查/添加审阅者。这样有助于我们避免评审队列的拥挤。
    
    ![Draft PR](/images/pr_draft.png)
    
2. **提供可复现的测试方案。** 请包含一个评估套件与仪表盘（用于比较 Stagehand 任务的模型性能与成本）（优先）或示例。如果我们无法运行任何能清楚突出你贡献的内容，就无法合并你的 PR。 
    1. 在 [`evals/tasks`](https://github.com/browserbase/stagehand/tree/main/evals/tasks) 中编写脚本，命名为 `someTask.ts` 
    2. 将脚本添加到 [`evals.config.json`](https://github.com/browserbase/stagehand/blob/main/evals/evals.config.json)，默认类别为 `combination`（如果你只测试 act/extract/observe，则使用 act/extract/observe）。
3. **添加 changeset。** 在 TS 中运行 `npx changeset` 或在 Python 中运行 `uvx changeset`，以添加一个会在下个版本的 `CHANGELOG` 中直接体现的 changeset。
    1. `patch` - 对终端用户无新增功能
    2. `minor` - 对终端用户有一定新增功能（新的函数参数、新导出类型等）
    3. `major` - 不应提交重大变更
