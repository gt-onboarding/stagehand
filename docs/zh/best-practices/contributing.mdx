<div id="general-workflow">
  ## 常规工作流程
</div>

加入[我们的贡献者名单](https://github.com/browserbase/stagehand/graphs/contributors)！

1. 开始前先讨论你计划的贡献。否则，你可能会面临把投入大量时间与精力的工作全部作废的风险。你可以在[Slack](https://stagehand.dev/slack) 上私信 Miguel 安排一次 1 对 1 通话。
2. 发起 Pull Request。Fork 此仓库，并按照[GitHub 的 Pull Request 创建说明](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/proposing-changes-to-your-work-with-pull-requests/creating-a-pull-request-from-a-fork)操作。这样我们的团队就能审阅你的贡献并留下评论。 
3. 等待审阅。我们会尽快处理你的贡献。如果 2–3 天仍未收到任何评论，请在[Slack](https://stagehand.dev/slack) 上私信 Miguel。
4. 合并到 `evals` 分支。为防止垃圾与滥用，我们不允许外部贡献者[通过 GitHub Actions 运行我们的 CI](https://github.com/browserbase/stagehand/blob/main/.github/workflows/ci.yml)。如果你的贡献通过初步筛查，我们会在其上运行我们的评估：
    1. 默认情况下，所有 PR 都会运行以下测试，你也可以在仓库中本地运行：
        1. Lint（`npm run lint`）- 运行 `prettier` 和 `eslint`。如果失败，通常可以运行 `npm run format` 修复部分简单的 lint 错误。
        2. Build（`npm run build`）- 通过 `tsup` 在 `dist/` 中完成 TS → JS 的检查与构建
        3. 端到端（`npm run e2e`）- 确定性的端到端 Playwright 测试，用于确保 [`stagehand.page`](http://stagehand.page) 与 `stagehand.context` 的基础 Playwright 功能完整性，以及与 Browserbase API 的兼容性
        4. 组合（`npm run evals category combination`）- 使用 `act`、`extract` 和 `observe` 的组合运行基于 AI 的端到端测试 
    2. 如果你对 `act`、`extract` 或 `observe` 本身做了任何更改，我们也可能运行特定的 act/extract/observe 评估，以确保现有功能不出现显著回退。
    
    ![CI](/images/CI.png)
    
5. 清理并合并到 main。一旦进入 `evals`，原始贡献者将无法继续修改代码。后续由 Stagehand 内部团队负责清理代码并将其合入 main。 

<div id="contribution-guidelines">
  ## 贡献指南
</div>

1. **使用草稿 PR。** 如果你的 PR 仍在进行中，请在开发期间将其转换为草稿（见下文），准备好后再标记为待评审并添加评审者。这样可以避免评审队列被占用。
    
    ![Draft PR](/images/pr_draft.png)
    
2. **提供可复现的测试方案。** 请附上一个评估（Evals，优先）或示例。如果无法运行能清晰体现你改动价值的内容，我们将无法合并你的 PR。
    1. 在 [`evals/tasks`](https://github.com/browserbase/stagehand/tree/main/evals/tasks) 中编写脚本 `someTask.ts`
    2. 将脚本添加到 [`evals.config.json`](https://github.com/browserbase/stagehand/blob/main/evals/evals.config.json)，默认分类为 `combination`（如果只测试 act/extract/observe，则分别使用 act/extract/observe）。
3. **添加 changeset。** 在 TS 项目中运行 `npx changeset`，或在 Python 项目中运行 `uvx changeset`，以添加一个 changeset；它会在下个版本的 `CHANGELOG` 中直接体现。
    1. `patch` - 对终端用户无新增功能
    2. `minor` - 对终端用户有一定新增功能（新增函数参数、公开类型等）
    3. `major` - 请勿提交重大变更
